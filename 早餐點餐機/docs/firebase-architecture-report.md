# 早餐點餐機 — Firebase 架構分析報告

## 1. 架構選擇

### 方案比較

| 方案 | 做法 | 優點 | 缺點 |
|------|------|------|------|
| A. 每店家一個 Firebase | 每個店家自己申請 Firebase 專案 | 完全獨立 | 設定麻煩，需為每個店家建專案 |
| B. 一個共用 Firebase | 所有店家共用一個專案，資料混在一起 | 設定最簡單 | 共用額度，一家用量大會影響全部，資料無隔離 |
| C. 一個共用 + 按店家分資料夾 | 同一個 Firebase，資料用 `shops/店家ID/` 區分 | 簡單 + 有隔離 | 額度共用但資料分開（推薦） |

### 推薦方案：C

所有店家共用一個 Firebase 專案，資料結構如下：

```
firestore:
  shops/
    ├── burger-shop-001/          ← 店家 A
    │     ├── menu (文件)          ← 菜單資料
    │     └── orders/ (子集合)     ← 訂單紀錄
    ├── egg-pancake-shop-002/     ← 店家 B
    │     ├── menu (文件)
    │     └── orders/ (子集合)
    └── ...
```

## 4. 店家使用流程

### 新店家加入
1. 你（管理者）在 Firebase 建立新的店家資料夾 `shops/{店家ID}/`
2. 幫店家上傳 menu.json
3. 產生店家專屬 QR Code（指向 `xxx.com/?shop={店家ID}`）
4. 店家不需要自己申請任何帳號

### 店家改菜單
1. 店家用手機/電腦進入後台 `admin.html?shop={店家ID}`
2. 修改菜單 → 即時寫入 Firestore
3. 客人掃 QR Code → 自動載入最新菜單

### 訂單紀錄
- 訂單存入 Firestore 的 `orders/` 子集合
- 後台可查詢歷史訂單
- 超過保留天數的訂單自動刪除

## 5. 安全性考量

| 項目 | 做法 |
|------|------|
| 店家資料隔離 | 每個店家只能讀寫自己的 `shops/{店家ID}/` |
| 後台密碼 | admin.html 需要密碼才能進入 |
| Firestore 規則 | 設定 per-shop 的讀寫權限 |
| QR Code | 只包含網址，不含敏感資訊 |

### Firestore 安全規則（概念）

```
match /shops/{shopId} {
  // 消費者只能讀菜單
  allow read: if true;
  
  // 只有管理員能寫菜單
  allow write: if request.auth != null;
  
  // 訂單：任何人可以寫（點餐），只有管理員可以讀（查看紀錄）
  match /orders/{orderId} {
    allow create: if true;
    allow read, write: if request.auth != null;
  }
}
```

| 決策 | 內容 |

| 超出額度 | 即用即付，500 家店月費約 NT$13 |
