Git / GitHub 基礎

理解版本控制的核心概念：什麼是倉庫（Repository）、怎麼提交（Commit）變化、怎麼推送到雲端。

💡 核心概念

Git 是什麼？

Git 是安裝在你電腦上的版本控制工具。它會記錄你每一次對檔案的修改，讓你可以「回到過去」。就像一個超強的「還原」按鈕。

GitHub 是什麼？

GitHub 是一個雲端平台，讓你把你電腦上的 Git 資料庫「上傳」到網路上，跟別人一起協作、備份你的程式碼。

比較GitGitHub
本質軟體工具（裝在電腦上）網站平台（在網路上）
功能管理檔案版本託管 + 協作 + 部署
需要網路？不需要需要

關鍵概念

當你執行 git init，Git 會在你的資料夾裡建立 .git/ 隱藏資料夾。這就是「Git 倉庫（Repository）」的本體。

✏️ 小測驗

1Git 的版本紀錄存在哪裡？

Git 的所有版本紀錄都存在專案資料夾內的 .git/ 隱藏目錄中，不需要網路就能運作。

2git add . 這個指令的作用是？

git add . 是「暫存（Stage）」的意思，把你修改過的檔案放進「待提交清單」，還沒有真正提交喔！

3Git 和 GitHub 的關係，最接近下列哪一個？

Git 是安裝在電腦上的工具，GitHub 是一個網站平台，用來在網路上託管你的 Git 倉庫。

4如果你不小心刪了一個檔案，用 Git 能救回來嗎？

只要檔案曾在某次 Commit 中被記錄過，就可以用 git checkout 或 git restore 從歷史中救回來！

5 .git/ 資料夾有什麼重要性？

.git/ 是 Git 倉庫的核心，裡面包含了所有的提交歷史、分支資訊、設定等。刪掉它就等於「這個資料夾不再是 Git 倉庫」。

🔀 Branch 分支

為什麼需要分支？

想象你在一條馬路上施工。如果直接在大馬路上施工，所有人都不能通行。分支就像繞一個便道，你在便道上施工，主線不受影響。

📋 GitHub Cheat Sheet

初始化倉庫（Initialize Repository）
在目前的資料夾建立一個新的 Git 倉庫
git init
複製遠端倉庫（Clone）
把 GitHub 上的倉庫下載到你的電腦
git clone https://github.com/使用者/專案.git
查看狀態（Status）
看哪些檔案被修改了、哪些還沒提交
git status
暫存所有變更（Add All）
把所有修改過的檔案加入待提交清單
git add .
提交（Commit）
把暫存區的變更拍一張快照
git commit -m "你的提交訊息"
推送（Push）
把本地的 Commit 上傳到 GitHub
git push origin main
拉取（Pull）
從 GitHub 下載最新的變更到本地
git pull origin main
建立並切換分支（Branch）
建立一個新分支並立刻切換過去
git checkout -b feature名稱
合併分支（Merge）
把指定分支的修改合併到目前的分支
git merge feature名稱
查看提交歷史（Log）
顯示所有 Commit 的紀錄
git log --oneline

Firebase 基礎

認識 Firebase 的核心服務：Hosting（網站託管）和 Firestore（雲端資料庫），以及它們的資料結構。

🔥 Firebase 是什麼？

🌐
Hosting（網站託管）
把你的 HTML/CSS/JS 網站放到 Google 的伺服器上，全世界都能訪問
🗄️
Firestore（資料庫）
雲端 NoSQL 資料庫，用來儲存和查詢資料
🔐
Authentication（驗證）
處理使用者登入/註冊，支援 Google、Email 等
⚙️
Functions（雲端函式）
在 Google 伺服器上執行後端程式碼

🌳 Firestore 資料結構

Firestore 的資料結構像俄羅斯娃娃：集合 → 文件 → 欄位 → 子集合 → …

📦 projects（集合 Collection）
📄 我的專案（文件 Document）
📝 createdAt: 2024-01-01
📝 updatedAt: 2024-06-15
📝 fileCount: 5
📁 files（子集合）
📄 index.html → { content, size }
📄 style.css → { content, size }

集合 vs 文件

集合（Collection）= 資料夾，用來放一群相關的文件。
文件（Document）= 檔案，裡面包含多個欄位（Field）。
一個文件裡面可以再放「子集合（Subcollection）」，無限嵌套。

🚀 Firebase Hosting 部署

什麼是 Hosting？

Firebase Hosting 幫你把網站檔案（HTML、CSS、JS）放到 Google 的全球 CDN 網路上，讓網站載入超快、免費、還自動加上 HTTPS。

# 部署步驟
1. npm install -g firebase-tools ← 安裝 Firebase 工具
2. firebase login ← 登入你的 Firebase 帳號
3. firebase init ← 初始化（會問你要用哪些功能）
4. firebase deploy --only hosting ← 上傳部署！

✏️ 小測驗

1Firestore 中，「集合（Collection）」和「文件（Document）」的關係最接近？

集合（Collection）就像資料夾，文件（Document）就像檔案。一個集合裡面可以有很多文件。

2Firebase Hosting 部署的指令是？

正確指令是 firebase deploy --only hosting，--only hosting 表示只部署網站部分。

3Firestore 中，一個文件（Document）裡面可以有什麼？

文件裡面可以有多種型態的欄位（文字、數字、布林值、陣列等），還可以再嵌套子集合，結構非常靈活。

4 firebase init 這個指令做了什麼？

firebase init 會問你幾個問題（要用哪些服務、public 資料夾名稱等），然後產生 firebase.json 設定檔。

5Firebase Hosting 的免費方案包含什麼？

Firebase Hosting 的 Spark（免費）方案包含：10 GB 儲存空間、每月 10 GB 下載量、自動 HTTPS 憑證、全球 CDN 加速。

Firebase 進階

學習 Security Rules（安全規則）保護你的資料庫，以及 Firestore 資料結構的最佳實踐。

🛡️ Security Rules（安全規則）

為什麼需要安全規則？

如果沒有安全規則，任何人都可以讀取、修改、刪除你資料庫裡的任何資料。Security Rules 就像是資料庫的「門禁系統」，決定誰能進來、能做什麼。

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // 🟢 允許所有人讀取 products 集合
    match /products/{productId} {
      allow read: if true;
      allow write: if false;
    }

    // 🟡 只有登入使用者能讀寫自己的資料
    match /users/{userId} {
      allow read, write: if request.auth != null
        && request.auth.uid == userId;
    }

  }
}

📝 規則語法解析

語法意思白話
allow read允許讀取可以看資料
allow write允許寫入可以新增/修改/刪除
if true條件永遠成立不限制，誰都可以
if false條件永遠不成立完全禁止
request.auth登入的使用者資訊看看有沒有登入
request.auth.uid使用者的唯一 ID確認是「誰」
resource.data目前資料庫裡的資料看看原本的內容

🏗️ 資料結構設計

扁平 vs 嵌套

Firestore 的設計建議「扁平化」—— 把資料攤平、減少嵌套層數。因為每多一層嵌套，讀取資料時就要多一次查詢。

命名前綴（Prefix）

為了避免不同專案的集合名稱衝突，建議加上專案縮寫作為前綴。例如：mpc-shortlinks、gps_中區建材行。

子集合 vs 欄位

如果一筆資料很小（幾個欄位），直接放在文件的欄位裡就好。只有當資料量很大或需要獨立查詢時，才用子集合。

⚙️ GitHub Actions

自動化工作流程

GitHub Actions 可以在你每次 Push 程式碼後，自動幫你執行指定的任務（例如：自動部署到 Firebase）。不用每次手動打指令！

# .github/workflows/deploy.yml
name: Deploy to Firebase
on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: FirebaseExtended/action-hosting-deploy@v0

⚖️ 兩種部署方式

方式手動部署自動部署

適合場景 個人小專案、測試用 團隊協作、正式上線
需要設定 Firebase 帳號即可 需要設定 GitHub Actions + Firebase Token
好處 簡單快速 一 Push 就上線，不用記指令

✏️ 小測驗

1當你執行 git push 後，GitHub Actions 什麼時候會自動幫你部署？

GitHub Actions 的觸發條件在 on.push.branches 裡設定，通常設定為只有推送到 main 分支才觸發部署。

2如果我想先在不影響正式站的情況下測試新功能，我應該怎麼做？

建立 Branch（分支）可以在不影響主線（main）的情況下自由開發。完成後再用 PR 合併回去。

3 firebase deploy 和 firebase deploy --only hosting 有什麼差別？

firebase deploy 會部署你專案中所有的服務（Hosting + Functions + Rules 等）。加上 --only hosting 只會部署 Hosting 部分。

4在「兩者整合」的流程中，GitHub 扮演什麼角色？

在整合流程中，GitHub 的角色是：1）存放原始碼並做版本控制；2）透過 Push 事件觸發 GitHub Actions 自動部署到 Firebase。

5你的網站部署到 Firebase 後，網址是什麼格式？

Firebase Hosting 預設給你的網址格式是 你的專案名.web.app 或 你的專案名.firebaseapp.com。你也可以設定自訂網域。
